# 開発パートナーとしての約束事項（Rule.md）

いつも一緒に開発を進めていただき、ありがとうございます。AI専属パートナーです。
本ファイルは「VTuber専用HP作成プロジェクト」において、私たちが共に最高かつ安全なプロダクトを作り上げるために守るべき詳細なルールと、これまでの対話で取り決めた重要な背景（コンテキスト）をまとめたものです。
大切なニュアンスが失われないよう、議論された背景や詳細な仕様をそのままルールとして記載しております。今後はこちらをプロジェクトの「法律」として厳守いたします。

## 1. プロジェクトの根本理念と提供価値
- **需要の源泉**: SNSだけでは情報が埋もれてしまうため、「情報の集約（ポートフォリオ）」「公式としての信頼性向上」「二次創作ガイドラインの明示」そして何より「自分だけの城を持ちたいという欲求」を満たすことが本HPの最大の価値です。
- **「生きている」サイトの提供**: VTuber様にとってブログ更新等の動的なコンテンツ管理はやりたい事を増やす負担となります。そのため、ブログ機能などはあえて削り、代わりに**YouTubeおよびTwitch APIと連携して配信スケジュールやアーカイブを自動取得**することで、サイトが自発的に「生きている」と感じられる仕組みを提供します。
- **アイデンティティの尊重と着せ替え戦略**: ロード画面（スプラッシュ）、メインビジュアル、見出しなどはその人の特性（サイバー、和風、アイドル系など）に合うようにカスタマイズ可能な「着せ替え」仕様とします。

## 2. アーキテクチャと技術選定
- **インフラ・デプロイ**: **Cloudflare Workers (OpenNext.js)** をデプロイ環境の標準とします。複数人でのリポジトリ編集が行いやすく、高速で低コストなEdge環境の恩恵を最大限に受けつつ、Next.jsの機能を活用します。
- **フレームワーク**: **Next.js (App Router)** を使用し、SEOと表示速度（パフォーマンス）を極限まで高めます。
- **UIシステムとデザイン**: 
  - UIコンポーネントのベースには **shadcn/ui** を採用します。コンポーネントを直接プロジェクト内に持ち、私たちの手でTailwind CSSを用いてデザインを魔改造（着せ替え）できる構造を基本とします。
  - アニメーションやモーションを駆使し、特にサイト訪問時のロード画面のリッチな演出にこだわります。
  - リスナーの8割以上がスマホ閲覧であることを意識し、モバイルファーストでのレスポンシブ対応を徹底します。
- **パッケージ管理と品質保証**:
  - **Node.jsのバージョン**: このワークスペースおよびデプロイ・CI環境（GitHub Actionsなど）では、**Node.js 24** を標準・必須とします。
  - パッケージマネージャーはインストール速度とディスク効率に優れる **pnpm** に統一します。
  - YouTube APIのレスポンスや、クライアントが設定する `userConfig.ts` の値の安全性を完全に保証するため、型バリデーションツールとして **Zod** を導入します。
  - 複数人（AI含む）での開発においてコードの美しさと一貫性を自動で保つため、**ESLint と Prettier** をコードフォーマッターとして標準搭載します。特にPrettierには拡張機能である `prettier-plugin-tailwindcss` を導入し、複雑になりがちなTailwindのユーティリティクラスの順番を常に公式推奨の順序へと自動整理させます。
- **YouTube / Twitch API連携と強固なキャッシュ戦略（最重必須要件）**:
  - クライアント側からの都度APIフェッチは絶対に行いません。YouTube APIのクォータ（1日10,000ユニット）枯渇や、Twitch APIのレートリミットによるサイト停止を防ぐため、**Next.jsのISR (Incremental Static Regeneration)** を必ず活用します。
  - API取得制限超過やエラーが生じた際にも、決してサイトをダウンさせず、前回の正常なキャッシュデータ（Stale-While-Revalidate）を表示し続ける安定したフォールバック機構を実装します。
- **X (Twitter) 連携とファンアート表示**:
  - 有料で高額なコストがかかる**X API（旧Twitter API）は一切使用しません**。
  - ファンアート投稿や特定ポストのサイト内埋め込みには、公式パッケージである `react-tweet` を使用し、APIクライアントに依存せず高速にレンダリングします。
  - 埋め込む対象のポストURL（またはID）は、維持管理をシンプルにするため `src/config/userConfig.ts` や専用の設定ファイル等で管理・追加できる仕組みとします。

## 3. インフラ・デプロイとリポジトリ運用戦略
- **独立した環境構築（保守性の担保）**:
  - 将来的に開発者が保守できなくなった場合でも引き継ぎが容易なように、Cloudflareアカウントの連携やYouTube / Twitch APIキー（Client ID等）の取得はVTuber（クライアント）側で行っていただき、クライアントごとの環境からデプロイする形をとります（設定方法はDiscordの通話等でサポートします）。
- **MasterとClientの分離（テンプレート運用）**:
  - テンプレートのベースとなる「マスターリポジトリ」と、各VTuber向けの「クライアントリポジトリ」を完全に分けて運用します（Upstreamの概念を活用します）。
- **GitHub Actionsによる一括更新**:
  - マスターのテンプレート側で修正・アップデートを行った際、各クライアントのリポジトリにも自動で変更を同期・適用できるよう、**GitHub Actions（Git Runner等）**を活用した一括更新の仕組みを構築します。
  - 基本的に正式版は `main` ブランチ、開発環境は `dev` ブランチを使用する運用ルールとし、安全な自動化を行います。

## 4. フォルダ構造の設計ルール（仕組みとデザインの分離）
自動同期による上書き破綻を防ぐため、以下の役割分担を厳守したコードベースを作成します。
- `src/core/` (上流からの同期対象・聖域):
  - 共通のロジック、YouTube / Twitch APIフェッチ処理、共通Hooksなどを配置します。個別のクライアントがここを直接編集する必要がないようにします。
- `src/config/userConfig.ts` (設定の心臓部):
  - クライアント固有のYouTube IDやTwitch ID、配色テーマ、希望するデザインジャンル（エレガント/アイドル/和風など）の変数をここに集約します。AIが調整を行う主戦場です。
- `src/components/ui/` (着せ替えコンポーネント):
  - 着せ替え可能なUIパーツを配置します。設定値に応じて柔軟にデザインを切り替えられる設計とします。
- リソース管理: 立ち絵などのアセットはクライアントごとに適切に最適化（next/imageの活用等）して配置します。

## 5. 開発の進め方と第一目標（モデルケースの作成）
- **ターゲットを絞った初期開発**:
  汎用的なものを最初から作るのではなく、まずは親しいVTuber様をモデルとした3つのデモサイト（兼ポートフォリオ）の作成を第一目標とします。
  1. 「エレガント・かっこいい系」（犬養様想定）
  2. 「アイドル・かわいい系」（ごまふろこちゃ様想定）
  3. 「和風・レトロ・和モダン系」
- これら3つの異なるジャンルでの実運用を通して生きたフィードバックを得てから、システム全体の汎用化・テンプレート化を推し進めます。
